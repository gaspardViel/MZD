# Context load vs cognitive load

Deux dimensions qui gouvernent le choix entre model-invoked et user-invoked. **Context load** : quantité d'information dont Claude a besoin pour exécuter le skill — élevé = OK pour model-invoked, les reference files absorbent la complexité. **Cognitive load** : charge mentale pour *toi* — savoir quand appeler le skill, comment le formuler. Les leading words dans la description réduisent ce load en créant un pont sémantique. **Règle side effects** : tout skill qui écrit, envoie ou modifie reste user-invoked — l'intention doit être délibérée. Lecture seule → model-invoked possible. **Le pont sémantique** : en user-invoked, les leading words dans la description permettent à Claude de reconnaître l'intention et de proposer l'invocation — tu confirmes. La friction diminue sans perdre le contrôle. `brief-hebdo-actif` : user-invoked (side effects Notion + Slack). `improve-skill` (leçon 14) : user-invoked (modifie des SKILL.md) mais description en leading words pour réduire le cognitive load.

**Implications** : Leçon 14 construit `improve-skill` — skill méta qui lit les learning-records et audite un skill cible contre les patterns appris.
