Leçon 13 · Claude Code × MZD

Context load vs cognitive load

Quand laisser Claude décider seul d'appeler un skill — et quand exiger que ce soit toi qui le fasses.

Où on en est

Tu sais écrire un skill robuste, compact, avec des barrières solides et sans failure modes. Il reste une décision de conception que tu n'as pas encore posée explicitement : qui déclenche ce skill ?

Il y a deux réponses possibles. La réponse change la façon dont tu écris la description — et elle détermine quels risques tu acceptes.

1. Model-invoked vs user-invoked

Model-invokedUser-invoked
Qui déclenche Claude reconnaît un pattern dans la conversation et appelle le skill seul Tu écris /nom-du-skill ou le demandes explicitement
Clé de déclenchement La description — Claude la lit pour décider si le skill correspond à l'intention Le nom du skill — tu dois le connaître et savoir quand l'appeler
Charge pour toi Faible — tu parles naturellement, Claude fait le lien Plus haute — tu dois décider toi-même du bon moment
Risque principal Claude peut se déclencher trop tôt, ou sur une mauvaise interprétation Tu peux oublier d'appeler le skill au bon moment

Ces deux modes coexistent dans un même projet. Tu choisis pour chaque skill lequel lui convient — en fonction de deux critères : le context load et le cognitive load.

2. Context load

Le context load est la quantité d'information dont Claude a besoin pour exécuter correctement le skill.

Un skill à context load élevé demande : des IDs de bases Notion, des IDs de canaux Slack, la structure des dossiers Drive, des règles métier sur les statuts. Sans ce contexte, Claude ne peut pas exécuter — ou il hallucine les IDs manquants.

La solution que tu connais déjà : les fichiers reference/*.md — une source de vérité stable que le skill lit en étape 0. Le context load élevé n'est pas un problème : c'est pour ça que les reference files existent.

Implication pour l'invocation : un skill à context load élevé convient au model-invoked — Claude gère lui-même la complexité du contexte, c'est exactement son rôle. Tu n'as pas à mémoriser les IDs Notion.

3. Cognitive load

Le cognitive load est la charge mentale que le skill fait peser sur toi : savoir quand l'appeler, comment le formuler, dans quel contexte il est approprié.

Un skill à cognitive load élevé nécessite que tu te souviennes de son nom exact, que tu saches à quel moment il est pertinent, et que tu le formules avec les bons paramètres. C'est de la friction — et elle s'accumule quand tu as 10 skills différents.

La solution : la description comme clé d'invocation. Si le skill est model-invoked et que sa description contient les mots que tu utilises naturellement, Claude fait le lien à ta place. Le cognitive load disparaît — tu parles comme tu penses, Claude convoque le bon outil.

4. La description comme clé d'invocation

Quand un skill est model-invoked, Claude lit toutes les descriptions de skills disponibles et cherche celle qui correspond le mieux à l'intention qu'il a détectée dans la conversation.

C'est là que les leading words de la leçon 10 prennent leur deuxième dimension. "Triage" dans la description ne sert pas seulement à décrire le skill — c'est le pont sémantique entre ta façon de parler et le déclenchement du skill.

Description sans ancrage
"Produit le brief hebdomadaire MZD et déclenche des actions sur les blocages détectés."

Si tu dis "qu'est-ce qui bloque cette semaine ?" — Claude ne fait pas forcément le lien. Les mots "brief" et "hebdomadaire" ne sont pas dans ta question.
Description avec leading words
"Weekly MZD triage — fetch context from Notion/Slack/Drive, triage items by urgency (🔴/🟡/🟢), dispatch alerts on blockers."

Si tu dis "fais un triage de ce qui bloque" — "triage" est dans la description. Claude propose le skill immédiatement.

5. Le garde-fou des side effects

Model-invoked est puissant — mais un skill avec des side effects (écriture dans Notion, message Slack, modification de fichiers) ne devrait jamais se déclencher sans intention explicite.

La règle Matt Pocock :

Side effects = user-invoked par défaut

Un skill qui lit peut être model-invoked — Claude reconnaît le besoin et fetche.
Un skill qui écrit, envoie, ou modifie doit être user-invoked — tu confirmes explicitement avant que Claude agisse.

Pourquoi ? Parce que "je veux un triage des tâches" et "envoie des alertes Slack à l'équipe" sont deux intentions très différentes. Claude peut reconnaître la première sans être autorisé à faire la seconde.

La solution pour brief-hebdo-actif : il a des side effects (Notion + Slack). Il reste donc user-invoked — tu l'appelles délibérément. Mais sa description contient des leading words pour qu'une fois que tu l'appelles, Claude l'exécute avec précision.

6. Matrice de décision

Profil du skillMode recommandéPourquoi
Lecture seule, context load élevé, fréquent Model-invoked Claude gère la complexité, tu parles naturellement
Side effects (écriture, envoi, modification) User-invoked L'intention doit être explicite avant d'agir
Skill rare, contexte très spécifique User-invoked Le déclenchement accidentel serait plus coûteux que la friction
Skill méta (améliore d'autres skills) User-invoked Modifie des fichiers — confirmation requise

7. Ce que ça implique pour improve-skill

Le skill qu'on va construire en leçon 14 — improve-skill — est un skill méta : il lit les learning records, analyse un skill cible, et propose (ou applique) des améliorations.

Son profil :

  • Context load élevé — il doit lire tous les learning records
  • Side effects potentiels — il peut modifier un SKILL.md
  • Usage délibéré — tu sais quand tu veux auditer un skill

Conclusion : user-invoked, mais avec une description en leading words pour que l'invocation soit naturelle quand tu la fais : 

description: >
  Audit a skill against learned patterns (leading words, completion
  criteria, failure modes) and propose targeted improvements.
  Run when a skill feels bloated, brittle, or hard to trigger.

"Audit", "leading words", "failure modes" — des mots que tu utilises dans ta conversation. Quand tu dis "j'ai l'impression que mon skill est trop verbeux" — Claude associe ce besoin au skill et te propose de l'invoquer. Tu confirmes. Il agit.

Quiz

Tu construis un skill qui lit les 5 derniers messages d'un canal Slack et en fait un résumé. Quel mode d'invocation est le plus adapté ?

Pourquoi brief-hebdo-actif reste user-invoked malgré une description optimisée en leading words ?

À quoi servent les leading words dans la description d'un skill user-invoked, si c'est quand même toi qui l'invoques ?

Ce qu'on a appris dans cette leçon

  • Model-invoked — Claude déclenche le skill en reconnaissant un pattern ; clé d'invocation = la description
  • User-invoked — tu déclenches explicitement ; la description guide l'exécution, pas l'invocation
  • Context load élevé → model-invoked OK — Claude absorbe la complexité à ta place
  • Side effects → user-invoked obligatoire — écriture, envoi, modification doivent être confirmés
  • Les leading words dans la description créent un pont sémantique même en user-invoked : Claude reconnaît l'intention et propose l'invocation, tu confirmes
  • improve-skill (leçon 14) sera user-invoked — context load élevé mais side effects sur les SKILL.md
