Leçon 3 · Claude Code × MZD

Skill intelligent : décision et action

Ajouter des branches conditionnelles et des actions d'écriture — transformer le brief passif en coordinateur actif.

Où on en est

Le skill brief-hebdo produit un résumé. C'est utile — mais un coordinateur ne se contente pas de lire. Il agit : il crée une tâche Notion quand un blocage est détecté, il envoie un message Slack quand une mission est sans responsable.

Cette leçon ajoute deux choses au pattern FETCH → REASON → ACT : une logique de décision (des branches) et des étapes d'action qui écrivent dans les vrais outils.

1. Les branches dans un skill

Une branche est un chemin conditionnel dans un skill. Selon ce que le fetch a trouvé, l'agent prend une route différente. Matt Pocock les documente comme des cas distincts dans le SKILL.md. 

## Étape 3 — Décider et agir

Pour chaque tâche dans le contexte :

**Branche A — Tâche bloquée depuis plus de 5 jours**
→ Appelle `notion-create-pages` pour créer un ticket "Alerte blocage"
   dans la base Tâches, lié à la tâche originale.
→ Appelle `slack_send_message` sur #info-generale avec un message
   concis identifiant le blocage et son responsable.

**Branche B — Mission sans responsable assigné**
→ Appelle `slack_send_message` sur le canal concerné
   pour demander qui peut prendre en charge.

**Branche C — Tout est en ordre**
→ Aucune action. Indique simplement dans le brief
   que la semaine est bien distribuée.
Règle des branches Chaque branche doit être mutuellement exclusive et exhaustive. L'agent doit pouvoir décider sans ambiguïté dans quelle branche il se trouve — sinon il choisit la branche la plus probable et peut se tromper.

2. Exemple réel — ce que le brief de cette semaine a révélé

Le brief du 26 juin 2026 a identifié plusieurs situations qui méritent une action automatique. Voici comment un skill intelligent les aurait traitées :

brief-hebdo (passif)
Produit un texte :

"Passation communication sans VSC : personne n'a encore repris les RS et la boîte mail animation."

Quelqu'un doit lire et agir manuellement.
brief-hebdo-actif (intelligent)
Détecte le blocage → crée automatiquement une tâche Notion "Passation com — responsable à désigner" et envoie un message Slack ciblé au trinôme Agathe / Gaspard / Sarah.

3. Le skill brief-hebdo-actif

---
name: brief-hebdo-actif
description: >
  Produit le brief hebdomadaire MZD ET déclenche des actions
  (tickets Notion, alertes Slack) sur les blocages détectés.
  Utilise chaque vendredi ou avant toute réunion de coordination.
---

## Étape 1 — Fetch les tâches Notion

Appelle `notion-query-database-view` sur la base "Tâches MZD".
Filtre : échéance dans les 7 prochains jours OU statut "En cours".
Continue quand chaque tâche (statut, assigné, date, dernière modif)
est dans le contexte.

## Étape 2 — Fetch les messages Slack

Appelle `slack_read_channel` sur #info-generale pour les 3 derniers jours.
Identifie : décisions prises, missions sans responsable, blocages exprimés.
Continue quand les fils pertinents sont dans le contexte.

## Étape 3 — Analyser et classer

Pour chaque tâche et chaque décision Slack, classe-la dans une
des trois catégories (uniquement depuis le contexte fetchté) :

- 🔴 BLOQUÉE : tâche "En cours" sans activité depuis plus de 5 jours,
  ou mission identifiée dans le CR CA sans responsable assigné
- 🟡 À SURVEILLER : tâche proche de l'échéance, responsable présent
  mais pas de mise à jour récente
- 🟢 EN ORDRE : avancement visible, responsable clair

## Étape 4 — Agir sur les blocages 🔴

Pour chaque item 🔴 :
a) Appelle `notion-create-pages` dans la base "Tâches MZD" :
   - Titre : "🔴 Alerte — [nom de la tâche bloquée]"
   - Propriétés : statut "À débloquer", lié à la tâche originale
b) Appelle `slack_send_message` sur #info-generale :
   - Message court (3 lignes max) : tâche bloquée, responsable,
     action attendue, lien Notion

Critère : chaque item 🔴 doit avoir généré un ticket ET un message
avant de passer à l'étape suivante.

## Étape 5 — Produire le brief final

Rédige le brief structuré (uniquement depuis le contexte — ne pas
inventer) :

- **Priorités** : les items 🟡 et 🟢 les plus importants (3 max)
- **Blocages traités** : liste des actions déclenchées à l'étape 4
- **À noter** : informations pertinentes des échanges Slack

Le brief est complet quand chaque item classé est mentionné
et quand les actions de l'étape 4 sont toutes confirmées.

4. Le garde-fou anti-spam

Un skill qui envoie des messages Slack automatiquement peut devenir irritant s'il est relancé plusieurs fois sur les mêmes données. Matt Pocock recommande d'ajouter un critère d'idempotence : 

## Avant l'étape 4 — Vérifier l'idempotence

Avant de créer un ticket ou d'envoyer un message, vérifie si
un ticket "🔴 Alerte — [nom]" existe déjà dans Notion
(appelle `notion-search` avec le titre exact).

Si le ticket existe et a été créé il y a moins de 3 jours :
→ Ne pas re-créer de ticket ni re-envoyer de message Slack.
→ Mentionne simplement le blocage existant dans le brief.
Pourquoi c'est important Sans ce garde-fou, relancer le skill deux fois dans la semaine génère deux tickets Notion et deux messages Slack pour le même blocage — ce qui crée du bruit et érode la confiance de l'équipe dans l'outil.

5. La structure FETCH → REASON → ACT au complet

FETCH   — Étapes 1 & 2  : lire Notion + Slack
REASON  — Étape 3       : classer les items (🔴 / 🟡 / 🟢)
CHECK   — Avant étape 4 : vérifier l'idempotence
ACT     — Étape 4       : créer tickets + envoyer messages
REPORT  — Étape 5       : produire le brief final

Le pattern s'est enrichi d'une étape CHECK entre REASON et ACT. C'est une bonne pratique pour tout skill qui écrit dans des outils partagés : vérifier avant d'agir.
