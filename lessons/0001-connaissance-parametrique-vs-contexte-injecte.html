Leçon 1 · Claude Code × MZD

Connaissance paramétrique vs contexte injecté

Pourquoi Claude ne "sait" pas ce qui se passe à la MZD — et comment y remédier avec des skills bien conçus.

Le problème fondamental

Claude est entraîné sur des données collectées jusqu'à une date de coupure. Après cette date, son savoir est figé — c'est ce qu'on appelle la connaissance paramétrique : ce que le modèle porte dans ses poids, gravé à l'entraînement.

Or la MZD produit des données nouvelles chaque jour : des tâches dans Notion, des messages dans Slack, des fichiers dans Drive. Ces données n'ont jamais été dans les données d'entraînement de Claude. Il ne les connaît pas. Il ne peut pas les deviner.

Règle n°1 — Matt Pocock Ne jamais faire confiance à la connaissance paramétrique du modèle pour des données qui changent dans le temps. Toujours injecter le contexte réel depuis les vrais outils.

Les deux types de connaissance dans un LLM

❌ Connaissance paramétrique
Ce que le modèle a appris à l'entraînement.

Figée. Peut être périmée. Ne contient rien sur les données internes de la MZD.

Utile pour : raisonner, écrire du code, résumer un texte qu'on lui donne.
✅ Contexte injecté
Ce que le skill place dans la fenêtre de contexte au moment de l'appel.

Frais. Vient des vrais outils (Notion, Slack, Drive) via les MCP tools.

Utile pour : raisonner sur l'état actuel de la MZD.

La puissance de Claude n'est pas dans ce qu'il "sait" — elle est dans sa capacité à raisonner à partir du contexte qu'on lui fournit. L'ingénierie des skills consiste précisément à construire ce contexte avec soin, avant de lui poser la moindre question.

Illustration : deux façons de demander le brief de la semaine

La mauvaise façon

# ❌ Le skill fait confiance au savoir paramétrique
"Claude, quelles sont les priorités de la MZD cette semaine ?"

Claude va inventer une réponse plausible — ou admettre qu'il ne sait pas. Dans les deux cas, c'est inutile. Il n'a pas accès aux tâches Notion, aux messages Slack, ni aux documents Drive de la MZD.

La bonne façon — le pattern "fetch-first"

# ✅ Le skill injecte le contexte avant de raisonner

## Étape 1 — Fetch les tâches Notion de cette semaine
→ mcp__Notion__notion-query-database-view(...)

## Étape 2 — Fetch les messages Slack récents
→ mcp__Slack__slack_read_channel(...)

## Étape 3 — Raisonner sur ce contexte réel
"Sur la base de ces tâches et messages, produis un brief
 de coordination pour l'équipe MZD."
Le pattern "fetch-first" Tout skill Claude Code qui travaille avec des données organisationnelles doit commencer par fetcher l'état réel depuis les outils, puis seulement raisonner. Ce séquençage est la colonne vertébrale de l'ingénierie des skills à la Matt Pocock.

Pourquoi ce pattern fonctionne

Les MCP tools (Notion, Slack, Drive) placent leurs résultats directement dans la fenêtre de contexte de Claude. À ce moment, Claude a "sous les yeux" des données réelles et récentes — il peut enfin raisonner de façon fiable.

C'est le même principe qu'un consultant externe à qui vous donnez le dossier avant de lui poser des questions : sans le dossier, il improvise ; avec le dossier, il raisonne.

La structure d'un bon skill MZD suit toujours cette logique : 

# Structure d'un skill MZD robuste

1. FETCH   — Lire l'état réel depuis Notion / Slack / Drive
2. REASON  — Demander à Claude de raisonner sur ce contexte
3. ACT     — Déclencher une action (créer une tâche, envoyer un message…)

Ce que le LLM fait bien (et qu'on garde)

La connaissance paramétrique reste utile pour :

  • Reformuler, synthétiser, rédiger — à partir du contexte injecté
  • Appliquer des raisonnements généraux (priorisation, résumé, classification)
  • Comprendre des instructions en langage naturel
  • Générer du code ou des structures à partir de specs

La combinaison gagnante pour la MZD : les outils fetchtent les données, Claude raisonne dessus.
