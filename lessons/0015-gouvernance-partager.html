Leçon 15 · Claude Code × MZD

Gouvernance partager

Rendre la structure MZD visible pour les bénévoles — du document Notion éparpillé au résumé partagé sur Slack.

Où on en est

Tu as construit les skills opérationnels : brief-hebdo-actif fetche le contexte chaque semaine, triage les blocages, et alerte l'équipe. Tes skills sont robustes — failure modes connus, critères de complétion explicites, invocation calibrée.

La mission MZD a quatre objectifs. Le premier — le brief hebdomadaire — est opérationnel. Le quatrième reste entier :

Objectif mission restant La structure de gouvernance est lisible : les instances (CA, réunions de pôle, journées bénévoles) sont connues et leur rôle est compris. Un bénévole qui arrive identifie clairement où sont les informations, comment s'investir, et qui contacter.

Cette leçon construit gouvernance-partager — le skill qui rend la gouvernance MZD visible.

1. Le problème invisible

Un nouveau bénévole qui rejoint la MZD fait face à un ensemble de questions sans réponse apparente :

  • Qui compose le CA ? Qui est le bureau ?
  • Quelles sont les instances de décision, et que décident-elles ?
  • Quand se réunissent les pôles ? Comment m'impliquer dans le mien ?
  • Qui contacter si j'ai une idée, un problème, une demande ?

Ces réponses existent quelque part dans Notion. Mais elles sont dispersées — une page par instance, un doc par pôle, une présentation datant de l'AG de l'an passé. Aucun bénévole n'a le temps de les rassembler.

Le skill gouvernance-partager fetche ces pages Notion, les synthétise, et produit un résumé clair — partageable sur Slack ou publié comme page d'accueil Notion pour les bénévoles.

2. Données stables vs données dynamiques

Ce skill est différent de brief-hebdo-actif. La différence n'est pas dans la technique — c'est dans la nature des données qu'il fetche.

brief-hebdo-actif gouvernance-partager
Type de données Dynamiques — tâches, blocages, événements de la semaine Stables — instances, rôles, structure de gouvernance
Fréquence Hebdomadaire — le contexte change chaque semaine Ponctuelle — quand la structure change (nouveau CA, nouveau pôle)
Durée de vie Une semaine — le brief est périmé le lundi suivant Plusieurs mois — le résumé reste valide jusqu'au prochain changement
Output Rapport éphémère — consommé immédiatement Document de référence — consulté dans le temps

Cette distinction est essentielle : un skill qui produit un document de référence a des exigences différentes d'un skill qui produit un rapport opérationnel. Le document doit être daté, sourcé, et ne contenir que ce qui reste vrai dans trois mois.

3. Le pattern "orient"

Dans les leçons précédentes, tu as utilisé le pattern triage : fetch → classer → alerter. C'est le pattern du brief hebdomadaire.

gouvernance-partager suit un pattern différent : orient. Il produit quelque chose que les bénévoles pourront consulter plutôt que consommer.

Pattern triage (brief-hebdo)
fetch → classer → alerter

Produit : liste de tâches urgentes, alertes Slack, actions à prendre cette semaine.

Durée de vie : 7 jours.
Périmé le lundi suivant.
Pattern orient (gouvernance-partager)
fetch → synthétiser → publier

Produit : vue d'ensemble structurée des instances, rôles, et points d'entrée pour les bénévoles.

Durée de vie : plusieurs mois.
Valide jusqu'au prochain changement structurel.

Le critère de complétion change : un brief hebdomadaire est complet quand tous les blocages sont triés. Un document de gouvernance est complet quand un bénévole qui le lit comprend la structure sans avoir besoin d'aide.

4. Conception du skill

Voici la structure complète de gouvernance-partager. Chaque étape découle des principes que tu as appris : context load injecté depuis les reference files, critère de complétion explicite, side effects déclarés, invocation user-invoked. 

---
name: gouvernance-partager
description: >
  Orient new volunteers — fetch MZD governance structure from Notion
  (CA, bureau, pôles, assemblées), synthesize instances, roles, and
  decision processes into a readable summary, and share it on Slack
  or update the Notion volunteer welcome page.
  Run when a new volunteer joins, when structure changes, or when
  governance needs to be made visible to the team.
argument-hint: "[canal Slack cible ou ID page Notion cible]"
---

## Objectif

Produire un résumé clair et sourcé de la gouvernance MZD à destination
des bénévoles — depuis les pages Notion officielles, sans connaissances
préalables requises.

## Étape 0 — Charger le contexte

Lire `reference/notion-structure.html` pour les IDs de pages.
Identifier les pages ciblées :
- Page Gouvernance (vue d'ensemble)
- Page CA + bureau
- Pages de chaque pôle
- Pages des instances récurrentes (AG, journées bénévoles)

Si `reference/notion-structure.html` est absent ou incomplet,
arrêter et signaler : "Référence Notion manquante — exécuter d'abord
le skill `fetch-reference-docs` pour cartographier la structure."

## Étape 1 — Fetcher la structure de gouvernance

Pour chaque instance identifiée dans les references, fetcher depuis Notion :
- Nom complet et acronyme
- Rôle et périmètre de décision
- Composition (membres, si disponibles)
- Fréquence et format des réunions
- Lien vers la page Notion source

Ne pas inférer ou compléter depuis la mémoire paramétrique.
Si une information est absente de Notion, la marquer [non renseigné].

## Étape 2 — Synthétiser

Produire un document structuré en 4 blocs :

**Bloc 1 — Vue d'ensemble**
Qui gouverne la MZD, comment les décisions sont prises, rôle des bénévoles.

**Bloc 2 — Les instances**
Pour chaque instance : nom, rôle, composition, fréquence, périmètre de décision.
Ordre : du plus stratégique (CA) au plus opérationnel (réunions de pôle).

**Bloc 3 — Comment s'impliquer**
Points d'entrée concrets pour un bénévole selon son profil :
- Je veux proposer une idée → qui contacter
- Je veux rejoindre un pôle → comment faire
- Je veux comprendre une décision → où chercher

**Bloc 4 — Sources et date**
Liste des pages Notion consultées.
Date de génération : {{date}}.
Mention explicite : "Ce document reflète la structure au {{date}}.
En cas de changement, relancer gouvernance-partager."

## Étape 3 — Publier

Si un canal Slack est fourni en argument : envoyer le résumé dans ce canal.
Si un ID de page Notion est fourni : mettre à jour cette page.
Si aucun argument : afficher le résumé dans la conversation et
demander confirmation avant de publier.

## Critère de complétion

✓ Toutes les instances de gouvernance tracées dans Notion sont mentionnées
✓ Chaque information est sourcée (lien Notion ou ID de page)
✓ Aucune structure ou rôle inventé — seulement ce qui est dans Notion
✓ Le document contient sa date de génération
✓ Un bénévole externe peut comprendre la structure sans aide supplémentaire

5. Le garde-fou de la stabilité

Parce que la gouvernance change rarement, le skill comporte un risque spécifique : produire un document daté mais non mis à jour. Si le CA change de composition et que le document reste inchangé sur Notion, les bénévoles lisent une gouvernance obsolète.

Règle stabilité → fraîcheur explicite Tout document produit par un skill "stable" doit mentionner sa date de génération et indiquer comment le renouveler. Plus un document est durable, plus il doit être explicitement daté.

Le critère de complétion inclut cette date pour cette raison. Ce n'est pas un détail cosmétique — c'est la signature de fraîcheur qui permet au lecteur de juger si le document est encore fiable.

Trois contenus à ne jamais inclure dans un document de gouvernance, parce qu'ils sont dynamiques :

  • L'agenda de la prochaine réunion de CA
  • Les tâches en cours dans chaque pôle
  • Les bénévoles actuellement actifs ou absents

Ces informations appartiennent au brief-hebdo-actif. Les deux skills sont complémentaires — l'un donne la structure, l'autre donne le contexte opérationnel de la semaine.

Quiz

Un nouveau bénévole vient d'être intégré et demande "comment fonctionne la MZD ?" Quel skill invoques-tu ?

Le CA vient d'élire un nouveau bureau lors de l'AG. À quel moment lances-tu gouvernance-partager ?

Parmi ces trois contenus, lequel a sa place dans le document produit par gouvernance-partager ?

Ce qu'on a appris dans cette leçon

  • Données stables vs dynamiques — la gouvernance change rarement ; ce n'est pas le même type de données que les tâches hebdomadaires. Le skill s'en charge différemment.
  • Pattern orient vs pattern triage — fetch → synthétiser → publier produit un document de référence durable, pas un rapport éphémère.
  • Critère de complétion étendu — pour un document stable, "complet" inclut la date de génération et les sources. Un lecteur doit pouvoir évaluer la fraîcheur du document lui-même.
  • Séparation des responsabilités — gouvernance-partager décrit la structure ; brief-hebdo-actif décrit le contexte opérationnel. Les deux sont complémentaires et ne doivent pas se chevaucher.
  • La règle de fraîcheur explicite — plus un document est durable, plus sa date doit être visible. L'ancienneté d'un document de référence est une information utile, pas un détail.
