Leçon 9 · Claude Code × MZD

Gestion des erreurs dans les skills

Que faire quand une source ne répond pas ou renvoie un résultat vide — et comment écrire des skills qui restent utiles même en conditions dégradées.

Où on en est

Le fetch parallèle de brief-hebdo-actif lance trois lectures simultanément : Notion, Slack, Drive. Jusqu'ici, on a supposé que les trois répondent correctement.

En pratique, ce n'est pas toujours le cas. Notion peut renvoyer zéro tâches (base vide ou filtre trop strict). Drive peut ne trouver aucun CR CA dans le dossier. Slack peut retourner un canal silencieux depuis 5 jours.

Ce ne sont pas des bugs — ce sont des états valides. La question est : que fait le skill dans ces cas ?

1. Les trois situations à gérer

SituationExemple concretDangereux ?
Résultat vide Notion renvoie 0 tâches, Drive trouve 0 fichiers dans le dossier ⚠️ Si le skill continue sans le signaler
Erreur technique L'outil échoue (timeout, ID invalide, permission refusée) ⚠️ Si le skill s'arrête sans explication
Données partielles Slack renvoie les messages mais le fil principal est vide ⚠️ Si le skill traite des données incomplètes comme complètes

Dans les trois cas, le vrai danger n'est pas l'erreur elle-même — c'est ce que le skill fait ensuite. Un skill mal écrit va soit s'arrêter silencieusement, soit pire, inventer du contexte pour combler le vide.

L'erreur la plus fréquente
Claude reçoit un résultat vide depuis Notion. Il n'y a pas d'instruction dans le SKILL.md pour ce cas. Claude raisonne quand même — et commence à "se souvenir" de tâches qu'il a vues lors de sessions précédentes, ou à en inventer de plausibles. C'est une hallucination silencieuse : le brief a l'air normal, mais il est basé sur rien.

2. La règle : signaler, ne jamais inventer

Un skill robuste suit une règle simple face à une source défaillante :

  1. Signaler — noter explicitement dans le brief que la source n'a pas répondu ou est vide
  2. Continuer avec ce qui est disponible — ne pas bloquer sur une seule source
  3. Ne jamais inventer — aucune donnée fabriquée, même "plausible"

Un brief avec une section "⚠️ Notion indisponible ce matin — données manquantes" est infiniment plus utile qu'un brief qui a l'air complet mais qui est basé sur des données inventées.

3. Comment l'écrire dans un SKILL.md

Il existe deux patterns selon le niveau de robustesse voulu.

Pattern minimal — la note de carence

Pour chaque source du fetch, ajouter une instruction de repli : 

## Étape 1 — Fetch (en parallèle)

Lance simultanément :

a) notion-query-database-view sur la base "Tâches MZD"
   Si résultat vide ou erreur :
   → noter "Notion : aucune tâche trouvée ou source indisponible"
   → continuer avec les autres sources

b) slack_read_channel sur #échanges-ca-équipe
   Si résultat vide ou erreur :
   → noter "Slack : canal silencieux ou source indisponible"
   → continuer avec les autres sources

c) search_files Drive dans le dossier CA 2026
   Si aucun résultat :
   → noter "Drive : aucun CR CA trouvé dans le dossier 2026"
   → continuer sans contexte CA

Continue quand toutes les tentatives sont terminées,
qu'elles aient réussi ou non.

La barrière change : on ne dit plus "Continue quand les trois réponses sont disponibles" — mais "Continue quand toutes les tentatives sont terminées". Le skill avance dans tous les cas.

Pattern complet — la section "Sources"

Pour un brief de qualité production, ajouter une section dédiée dans l'output final : 

## Étape 5 — Produire le brief final

Commence le brief par une section "Sources" :

✅ Notion — [N] tâches lues
✅ Slack — [N] messages lus sur les 3 derniers jours
⚠️ Drive CR CA — aucun fichier trouvé dans le dossier 2026
   → section "Contexte CA" omise

Ensuite les sections habituelles.
Si une source est manquante, sa section correspondante est
omise du brief avec une mention explicite.

Cette section "Sources" en tête de brief sert de tableau de bord rapide : au premier coup d'œil, tu sais sur quoi le brief est basé.

4. Résultat vide ≠ erreur technique

Il y a une distinction importante à faire dans le SKILL.md :

CasSignificationQue noter dans le brief
Notion renvoie 0 tâches Soit tout est en ordre, soit le filtre est trop strict "Notion : 0 tâche correspondant aux critères" (état valide possible)
Notion échoue (erreur) Problème de connexion ou d'autorisation "Notion : source indisponible — données non lues"
Drive : 0 fichier dans le dossier 2026 Soit pas encore de CR CA cette année, soit mauvais ID "Drive : aucun CR CA trouvé — vérifier l'ID du dossier dans drive-structure.md"
Slack : 0 message sur 3 jours Canal silencieux — information valide en soi "Slack : canal #échanges-ca-équipe silencieux depuis 3 jours"

Un canal Slack silencieux depuis 3 jours, ce n'est pas une erreur — c'est une information. Le brief doit le dire explicitement plutôt que de l'ignorer ou de combler le vide.

5. Mettre à jour brief-hebdo-actif

On va ajouter les instructions de repli à l'étape 1 et la section "Sources" à l'étape 5. Le reste du skill ne change pas. 

## Étape 1 — Fetch (en parallèle)

Lance simultanément :

a) notion-query-database-view sur la base "Tâches MZD"
   → ID dans reference/notion-structure.md
   Filtre : échéance ≤ 7j OU statut "En cours"
   Si 0 résultat ou erreur → noter, continuer

b) slack_read_channel sur #échanges-ca-équipe
   → ID dans reference/slack-channels.md
   Pour les 3 derniers jours
   Si 0 message ou erreur → noter, continuer

c) Dernier CR CA Drive :
   search_files dans le dossier CA de l'année en cours
   → ID du dossier dans reference/drive-structure.md
   Si 0 résultat → noter "aucun CR CA trouvé", continuer sans c)

Continue quand toutes les tentatives sont terminées.

## Étape 5 — Produire le brief final

Commence par une section "Sources" :
✅/⚠️ Notion — [N tâches] ou [indisponible]
✅/⚠️ Slack — [N messages / 3 jours] ou [silencieux] ou [indisponible]
✅/⚠️ Drive CR CA — [date du CR lu] ou [aucun trouvé]

Puis les sections habituelles.
Omettre une section si sa source est manquante — avec mention explicite.

6. La règle qui sous-tend tout ça

La gestion des erreurs dans les skills, c'est une application directe du principe de la leçon 1 : Claude ne sait rien de la MZD par lui-même.

Quand une source est vide, Claude n'a pas de données MZD "de secours" dans sa mémoire. Il n'a que ce qui a été fetchté dans cette session. Un skill qui ne gère pas les erreurs laisse Claude dans le vide — et le vide, pour un LLM, produit des hallucinations.

Le fil rouge depuis la leçon 1
Leçon 1 : inject context before reasoning.
Leçon 9 : if the context fetch fails, say so — don't reason on nothing.

C'est la même règle, appliquée aux cas limites.

Ce qu'on a appris dans cette leçon

  • Trois situations à gérer — résultat vide, erreur technique, données partielles. Aucune n'est un bug en soi ; c'est ce que le skill fait ensuite qui compte.
  • Signaler, ne jamais inventer — un brief incomplet mais honnête est meilleur qu'un brief complet et halluciné
  • Résultat vide ≠ erreur — un canal silencieux est une information ; 0 tâche peut être un état valide
  • La barrière "tentatives terminées" — le skill avance même quand une source échoue, sans bloquer sur elle
  • La section "Sources" — tableau de bord en tête de brief : ✅/⚠️ pour chaque source, avant toute analyse
  • Le fil rouge depuis la leçon 1 — si le contexte n'a pas été fetchté, ne pas raisonner dessus. Les erreurs de fetch sont le même principe appliqué aux cas limites.
