Ancrer les skills dans la réalité MZD

Reference docs — documenter la structure Notion et Slack une fois, citer partout, ne plus jamais hardcoder un ID dans un skill.

Où on en est

Tu as maintenant quatre skills opérationnels : /brief-hebdo, /brief-hebdo-actif, /coordination-mzd et /cloture-semaine. Et si tu regardes de près leurs fichiers SKILL.md, tu remarques quelque chose. 

## Étape 2 — Fetch les messages Slack récents

Appelle slack_read_channel sur le canal #info-generale
(C02085Z5CHL), les 3 derniers jours.

C02085Z5CHL. Un ID opaque, gravé en dur dans le texte. Si demain ce canal est renommé, archivé, ou remplacé par un autre — ce skill est cassé. Et il faut se souvenir de l'aller corriger à la main.

Ce n'est pas un problème de code. C'est un problème de connaissance tribale : l'ID vit dans les skills au lieu de vivre dans un endroit prévu pour ça.

1. Chaque ID hardcodé est un bug futur

La règle d'or de l'ingénierie des skills Matt Pocock s'applique ici : ce que Claude doit chercher dans le monde réel ne doit pas être fixé dans le texte du skill.

La leçon 1 a établi que la connaissance figée est dangereuse — le modèle peut inventer des données qu'il ne connaît pas. La même logique s'applique à la structure de ton organisation.

Danger
C02085Z5CHL dans un SKILL.md, c'est de la connaissance figée sur ta propre infrastructure. Elle se désynchro­nise dès que tu touches à Slack.

Voici ce qui peut arriver sans reference docs :

  • Le canal #info-generale est renommé → skill silencieusement cassé
  • Une nouvelle base Notion est créée → les skills ignorent son existence
  • Un nouveau membre du CA veut écrire un skill → il ne sait pas où trouver les IDs
  • Tu relies un skill à Slack six mois plus tard → tu dois retrouver l'ID à la main

La solution n'est pas de memoriser les IDs — c'est de les documenter une fois dans un fichier prévu à cet effet, et de citer ce fichier depuis chaque skill qui en a besoin.

2. Les reference docs

Un reference doc est un fichier markdown qui documente la structure réelle de ton organisation : canaux Slack, bases Notion, dossiers Drive. Il vit dans le dossier reference/ du repo.

Sans reference docs
L'ID C02085Z5CHL est dans brief-hebdo.
L'ID C02085Z5CHL est dans brief-hebdo-actif.
L'ID C02085Z5CHL est dans cloture-semaine.

3 endroits à corriger si le canal change. Et probablement d'autres qu'on a oubliés.
Avec reference docs
reference/slack-channels.md :
#info-generale → C02085Z5CHL

Tous les skills citent ce fichier.
1 seul endroit à mettre à jour.
Claude lit la référence à chaque exécution.

La clé : Claude ne mémorise pas les IDs entre les sessions. À chaque invocation du skill, il relit les reference docs depuis le repo — comme il relit les SKILL.md. C'est du contexte injecté, pas de la connaissance figée.

Pattern : document once, cite everywhere

Chaque reference doc est la source de vérité. Les skills ne contiennent que le nom humain du canal ou de la base, et une note pointant vers le fichier de référence.

4. Citer un reference doc depuis un SKILL.md

Le changement est simple : dans chaque étape qui utilise un ID, on remplace l'ID par son nom humain et on ajoute une note pointant vers le reference doc.

Avant — ID hardcodé
Appelle slack_read_channel
sur C02085Z5CHL,
les 3 derniers jours.
Après — nom cité
Appelle slack_read_channel
sur #info-generale.
→ ID dans reference/slack-channels.md

Claude lira reference/slack-channels.md au moment de l'exécution pour résoudre le nom en ID. Si le fichier est à jour, le skill fonctionne — même si l'ID a changé.

Règle : quand mettre à jour les reference docs ?

Quand un évènement réel se produit dans ton organisation :

  • Un nouveau canal Slack est créé → l'ajouter à reference/slack-channels.md
  • Une nouvelle base Notion est mise en place → l'ajouter à reference/notion-structure.md
  • Un canal est archivé → mettre "Archivé" dans le fichier, ne pas supprimer (historique)
  • Un ID change (migration Notion) → corriger à un seul endroit
À ne pas faire
Ne pas laisser un skill "découvrir" les IDs à chaque exécution via un search Slack ou Notion — c'est lent, coûteux, et fragile. Les reference docs remplacent cette découverte par une documentation explicite et maintenue.
