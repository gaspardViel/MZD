Leçon 4 · Claude Code × MZD

Composer des skills ensemble

Router skill et orchestration — passer de deux skills isolés à un système de coordination MZD complet.

Où on en est

Tu as deux skills opérationnels : /brief-hebdo (passif) et /brief-hebdo-actif (crée des tickets + envoie des alertes Slack). La question qui se pose naturellement : quand utiliser lequel ? Et si d'autres skills s'ajoutent à l'avenir, comment s'y retrouver ?

Matt Pocock a une réponse à ça : le router skill.

1. Le problème : charge cognitive vs charge de contexte

Chaque skill que tu crées te demande de mémoriser son nom et son bon moment d'usage. Matt Pocock appelle ça la cognitive load — la charge que tu portes en tête en tant qu'humain. Elle augmente avec chaque skill user-invoked.

En face, il y a la context load : le coût qu'un skill model-invoked impose à chaque échange en chargeant sa description dans la fenêtre de contexte de Claude.

Sans router — cognitive load élevée
Tu dois te souvenir :

/brief-hebdo → vendredi matin, lecture seule
/brief-hebdo-actif → vendredi midi, actions
/check-programmation → à venir
/alerte-bail → à venir

4 skills à retenir, et ça grossit.
Avec router — un seul point d'entrée
Tu retiens un seul nom :

/coordination-mzd

Le router te demande ce dont tu as besoin et t'oriente vers le bon skill. Un skill à retenir, quel que soit le nombre de skills derrière.

2. Le router skill

Un router skill est un skill user-invoked dont le seul rôle est de cartographier les autres skills et d'indiquer lequel atteindre selon la situation. Il pointe vers les autres — il ne les exécute pas lui-même.

Ce que le router ne fait pas Un router skill ne peut pas déclencher automatiquement un autre skill user-invoked — seul l'humain peut le faire. Le router fournit la carte ; toi tu appuies sur la gâchette.

L'exemple parfait dans ce repo : /ask-matt. C'est un router pur : il liste les flows disponibles et te dit lequel utiliser selon ta situation. Il ne code pas, ne génère pas de PRD — il navigue.

4. Les deux types de composition

Matt Pocock distingue deux façons de composer des skills :

Type A — Router (navigation)

Un skill qui pointe vers d'autres skills selon la situation. C'est coordination-mzd. L'humain reste aux commandes : il lit la carte et choisit.

Type B — Séquence orchestrée

Un skill qui exécute une chaîne d'actions dans l'ordre. Par exemple, un skill clôture-semaine qui : 

---
name: cloture-semaine
description: >
  Clôture de semaine MZD complète : brief actif + résumé prog
  + message récapitulatif sur Slack. À lancer chaque vendredi à 17h.
---

## Étape 1 — Exécuter le brief hebdomadaire actif
Exécute toutes les étapes de `/brief-hebdo-actif` :
fetch Notion, fetch Slack, classifier, vérifier idempotence,
créer tickets 🔴, envoyer alertes.
Continue quand le brief est produit et les actions confirmées.

## Étape 2 — Fetch la todo liste programmation
Appelle `notion-fetch` sur la Todo list programmation.
Identifie les tâches ouvertes avec deadline dans les 14 prochains jours.
Continue quand la liste est dans le contexte.

## Étape 3 — Composer le message de clôture
Sur la base du brief (étape 1) et des tâches prog (étape 2),
rédige un message Slack de clôture de semaine :

- 🗓️ **Semaine prochaine** : 3 priorités max
- 🔴 **Blocages traités** : résumé des alertes envoyées
- 📢 **Com à venir** : publications programmées ou à programmer

Message : 10 lignes maximum, ton direct et actionnable.

## Étape 4 — Envoyer sur Slack
Appelle `slack_send_message` sur #info-generale avec le message
composé à l'étape 3.
Critère : confirmation d'envoi reçue.
Attention : coût de la séquence orchestrée Une séquence qui enchaine plusieurs fetches et plusieurs actions peut être longue et consommer beaucoup de contexte. Garde les séquences courtes (3-4 étapes max) et décompose les sequences longues en plusieurs skills séparés avec un router.

5. Router ou séquence — comment choisir ?

Situation                              → Choix
─────────────────────────────────────────────────────────
Tu veux naviguer entre plusieurs       → Router skill
  options selon le contexte

Tu veux exécuter une séquence fixe     → Séquence orchestrée
  toujours dans le même ordre

Tu accumules > 5 skills user-invoked   → Ajoute un router
  et tu commences à les oublier

Les étapes d'une séquence varient      → Router + skills séparés
  selon les données fetchées
