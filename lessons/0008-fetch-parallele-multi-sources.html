Leçon 8 · Claude Code × MZD

Fetch parallèle et multi-sources

Lire Notion, Slack et Drive simultanément plutôt que l'un après l'autre — et enrichir brief-hebdo-actif avec la troisième source.

Où on en est

Tu as trois sources de contexte MZD : Notion (tâches), Slack (échanges), Drive (documents). Les skills actuels lisent ces sources une par une — d'abord Notion, puis Slack, puis (bientôt) Drive.

Ce chaînage séquentiel a un coût invisible : si chaque lecture prend 3 secondes, trois lectures prennent 9 secondes. En parallèle, les mêmes trois lectures prennent 3 secondes.

Mais le parallélisme a une règle stricte — et la comprendre, c'est comprendre comment écrire des skills robustes.

1. Le problème du fetch séquentiel

Voici comment brief-hebdo-actif fonctionne aujourd'hui : 

Étape 1 — Fetch Notion  (attend la réponse)
     ↓
Étape 2 — Fetch Slack   (attend la réponse)
     ↓
Étape 3 — Analyser

La lecture Slack ne peut commencer que quand la lecture Notion est terminée. Ce n'est pas une contrainte technique — c'est juste la façon dont le skill est écrit. Notion et Slack sont indépendants : l'un n'a pas besoin du résultat de l'autre pour démarrer.

Avec Drive en troisième source, le problème s'amplifie : 

Étape 1 — Fetch Notion  (3s)
     ↓
Étape 2 — Fetch Slack   (3s)
     ↓
Étape 3 — Fetch Drive   (3s)
     ↓
Étape 4 — Analyser

Total : ~9 secondes d'attente

2. La règle : parallèle pour les lectures, séquentiel pour les actions

Le parallélisme n'est pas toujours possible. La règle est simple :

Type d'opérationParallélisable ?Pourquoi
Fetch Notion, Slack, Drive ✅ Oui Lecture seule — aucun effet de bord, aucune dépendance entre eux
Analyser (étape 3) ❌ Non Doit attendre que toutes les sources soient lues
Créer un ticket Notion ❌ Non Doit attendre l'analyse pour savoir si un ticket est nécessaire
Envoyer un message Slack ❌ Non Dépend du ticket Notion créé à l'étape précédente (lien à inclure)
Vérifier l'idempotence (notion-search) ✅ Oui (entre items) Chaque vérification est indépendante des autres
La règle à retenir
Deux opérations peuvent être parallèles si et seulement si aucune n'a besoin du résultat de l'autre pour s'exécuter. Les fetches de sources indépendantes : toujours parallèles. Les actions qui dépendent d'une analyse : toujours séquentielles.

3. Écrire le parallélisme dans un SKILL.md

Claude sait exécuter des opérations en parallèle — il suffit de l'écrire explicitement dans le SKILL.md. Deux formulations fonctionnent :

Séquentiel (lent)
## Étape 1 — Fetch Notion
Appelle notion-query-database-view.
Continue quand tu as les tâches.

## Étape 2 — Fetch Slack
Appelle slack_read_channel.
Continue quand tu as les messages.
Parallèle (rapide)
## Étape 1 — Fetch (en parallèle)

Lance simultanément :
- notion-query-database-view
  sur la base Tâches MZD
- slack_read_channel
  sur #échanges-ca-équipe
- search_files (Drive)
  dossier CA 2026

Continue quand les trois
réponses sont disponibles.

Le mot "simultanément" ou "en parallèle" suffit à signaler à Claude qu'il peut lancer toutes les opérations listées sans attendre chacune. La barrière "Continue quand les trois réponses sont disponibles" garantit que l'analyse n'commence qu'une fois tout le contexte en main.

5. Le SKILL.md complet après mise à jour

Voici à quoi ressemble brief-hebdo-actif après les changements. Les étapes d'action (3 à 6) restent identiques — seul le fetch change. 

---
name: brief-hebdo-actif
description: >
  Produit le brief hebdomadaire MZD ET déclenche des actions
  (tickets Notion, alertes Slack) sur les blocages détectés.
  Utilise chaque vendredi ou avant toute réunion de coordination.
---

## Contexte à lire avant de commencer

Lis ces trois fichiers pour résoudre les noms en IDs :
- reference/slack-channels.md
- reference/notion-structure.md
- reference/drive-structure.md

## Étape 1 — Fetch (en parallèle)

Lance simultanément :

a) notion-query-database-view sur la base "Tâches MZD"
   → ID dans reference/notion-structure.md
   Filtre : échéance dans les 7 prochains jours OU statut "En cours"

b) slack_read_channel sur #échanges-ca-équipe
   → ID dans reference/slack-channels.md
   Pour les 3 derniers jours

c) Dernier CR CA Drive :
   - search_files({ query: "parentId = 'ID_DOSSIER_2026'" })
     → ID_DOSSIER_2026 dans reference/drive-structure.md
   - trier par modifiedTime desc, prendre le premier résultat
   - read_file_content sur ce fichier

Continue quand les trois réponses sont disponibles dans le contexte.

## Étape 2 — Analyser et classer

Pour chaque tâche Notion et chaque décision Slack, classe dans :
- 🔴 BLOQUÉE : tâche "En cours" sans activité > 5 jours,
  ou mission sans responsable
- 🟡 À SURVEILLER : proche de l'échéance, pas de mise à jour récente
- 🟢 EN ORDRE : avancement visible, responsable clair

Continue quand chaque item est classé.

## Étape 3 — Vérifier l'idempotence avant d'agir

Pour chaque item 🔴, appelle notion-search "🔴 Alerte — [nom]".
Si ticket créé il y a moins de 3 jours → ne pas recréer, marquer "déjà traité".

## Étape 4 — Agir sur les blocages 🔴

Pour chaque item 🔴 non déjà traité :
a) notion-create-pages dans "Tâches MZD" — titre "🔴 Alerte — [nom]"
b) slack_send_message sur #échanges-ca-équipe — 3 lignes max

## Étape 5 — Produire le brief final

Sections :
- Priorités (🟡 et 🟢, 3 max, avec responsable et échéance)
- Blocages traités (actions étape 4 ou "déjà traité")
- À noter (échanges Slack pertinents)
- Contexte CA (date + 2 décisions clés du dernier CR CA Drive)

6. Pourquoi ça change quelque chose en pratique

Ce n'est pas juste une question de vitesse. Le fetch parallèle change la façon dont on pense l'architecture d'un skill.

AvantAprès
Ajouter une source = ajouter une étape séquentielle Ajouter une source = ajouter une ligne dans l'étape de fetch
Le brief attendait 6–9 secondes avant d'analyser Le brief attend 3 secondes (le temps de la source la plus lente)
3 sources → 3 "Continue quand…" séparés 3 sources → 1 seule barrière "Continue quand les trois réponses…"
Difficile à étendre (4ème source = 4ème étape) Facile à étendre (4ème source = 4ème ligne dans le fetch)
Ce que le parallélisme ne résout pas
Le parallélisme accélère le fetch, pas le raisonnement. Si les trois sources renvoient des données volumineuses, Claude doit toujours traiter tout ce contexte avant d'analyser. La bonne pratique reste de filtrer en amont : pageSize limité, dates récentes, statuts pertinents — pour ne fetcher que ce dont le skill a besoin.

Ce qu'on a appris dans cette leçon

  • Fetch séquentiel vs parallèle — le temps total est la somme des fetches en séquentiel, le maximum en parallèle
  • La règle de parallélisme — deux opérations sont parallélisables si aucune n'a besoin du résultat de l'autre
  • Lectures = parallèles, actions = séquentielles — les fetches Notion/Slack/Drive sont indépendants ; créer un ticket ou envoyer un message dépend toujours de ce qui précède
  • "Simultanément" dans le SKILL.md — le mot suffit à déclencher l'exécution parallèle, suivi d'une barrière explicite
  • brief-hebdo-actif enrichi — trois sources (Notion + Slack + Drive CR CA) lues en une seule étape parallèle, brief final avec section "Contexte CA"
