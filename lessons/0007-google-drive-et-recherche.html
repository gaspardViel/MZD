Leçon 7 · Claude Code × MZD

Google Drive et recherche

Lire les documents partagés de la MZD depuis les skills — et savoir quand chercher dans Drive plutôt que dans Notion.

Où on en est

Tu as un système complet : Notion pour les tâches structurées, Slack pour les échanges d'équipe, des skills qui orchestrent tout ça, et des crons pour déclencher automatiquement.

Il manque un troisième outil de lecture : Google Drive. La MZD y stocke ses comptes-rendus CA, ses documents de gouvernance, ses plannings bénévoles. Autant de contexte que les skills ne peuvent pas lire — tant qu'on ne leur apprend pas comment chercher.

1. Drive vs Notion — quand utiliser quoi

La confusion la plus fréquente : tout ce qui n'est pas dans Slack est censé être dans Notion. Ce n'est pas le cas pour la MZD. Deux outils coexistent, avec des rôles distincts.

BesoinOutilPourquoi
Tâches, statuts, filtres Notion Base de données structurée, requêtable par statut / date
CR CA 2026 et après Notion Migration récente — les nouveaux CRs sont des pages Notion
CR CA 2025 et avant Drive Historique dans les dossiers CA, format Google Docs
Documents longs, présentations Drive Google Docs / Slides / Sheets — format natif
Planning bénévoles Drive Google Sheets partagé avec toute l'équipe
Gouvernance des pôles Drive Dossier "Gouvernance Partagée" piloté depuis maisonduzd@gmail.com
Règle pratique
Quand un skill a besoin de contexte et qu'il ne trouve rien dans Notion, la prochaine question est : "est-ce que c'est dans Drive ?" Avant cette leçon, la réponse était toujours "je ne sais pas". Après cette leçon, Claude sait où chercher.

2. Les outils Drive MCP

Trois outils composent le trio de lecture Drive. Ils fonctionnent de la même façon que les outils Notion : tu les appelles, Claude lit, tu raisonnes sur ce qu'il trouve.

OutilUsageParamètre clé
search_files Trouver des fichiers par titre, dossier parent, date, type query — syntaxe Drive query
get_file_metadata Vérifier qu'un fichier existe et obtenir ses infos (titre, propriétaire, date) fileId
read_file_content Lire le contenu texte d'un Google Doc ou Sheet fileId

La syntaxe de recherche Drive

search_files utilise une syntaxe de requête spécifique. Ce ne sont pas des mots-clés libres — c'est un mini-langage de filtres. 

// Chercher par titre
search_files({ query: "title contains 'Compte Rendu CA'" })

// Chercher dans un dossier spécifique
search_files({ query: "parentId = '1flx8G4bDXDz-PpEu47RulpZwN_XUTdqq'" })

// Chercher un type de fichier
search_files({ query: "mimeType = 'application/vnd.google-apps.document'" })

// Combiner : titre + dossier parent
search_files({ query: "title contains 'CA' and parentId = '1BYIZfmSNz-FZwmdh2RfggnFOuf28tEpM'" })

// Chercher les fichiers modifiés récemment
search_files({ query: "modifiedTime > '2026-01-01T00:00:00Z'" })
Piège courant
fullText contains 'mot' cherche dans le contenu des fichiers — c'est lent et peu fiable pour les Google Docs. Préfère toujours title contains quand tu cherches par nom. Utilise read_file_content après pour lire le contenu.

3. La reference doc Drive

On vient de créer reference/drive-structure.md avec la carte réelle de la MZD — les IDs vérifiés des dossiers CA, du dossier Gouvernance Partagée, du planning bénévoles.

C'est le même principe que reference/notion-structure.md : centraliser les IDs et les patterns pour que les skills n'aient pas à les deviner. Un ID Drive qui change (si le fichier est déplacé) n'invalide qu'un endroit — la reference doc — pas tous les skills.

Ce que contient drive-structure.md

  • Dossiers CA par année — IDs des dossiers 2024, 2025, 2026 et nomenclature des fichiers (YYYY_MM_DD CA)
  • Gouvernance Partagée — dossier MZD et documents clés (Cadrage gouvernance, Tableaux Gouvernance)
  • Créneaux bénévoles — le Google Sheet partagé avec toute l'équipe
  • Tableau Drive / Notion — pour savoir où chercher quoi

4. Le pattern "search before read"

Contrairement à Notion où chaque base a un ID stable qu'on cite directement, les fichiers Drive peuvent être déplacés ou renommés. Le pattern robuste : chercher d'abord, puis lire.

Fragile
Hardcoder l'ID du fichier directement dans le SKILL.md.

Si le fichier est déplacé → le skill échoue sans explication.
Robuste
  1. Utiliser l'ID du dossier depuis drive-structure.md
  2. search_files(parentId = ID_DOSSIER)
  3. Choisir le fichier dans les résultats
  4. read_file_content(fileId)

Exception : les documents stables et clés (Tableaux Gouvernance, Créneaux bénévoles) peuvent avoir leur ID cité directement dans drive-structure.md, comme on le fait pour les bases Notion. Mais même là, on lit d'abord le fichier de référence, pas le skill.

5. Créer le skill cherche-cr-ca

Un exemple concret : un skill qui retrouve le dernier CR CA dans Drive et en extrait les points de décision. 

# SKILL.md — cherche-cr-ca

## Quand utiliser ce skill
Quand on veut retrouver les décisions du dernier Conseil d'Administration
enregistrées dans Drive (CRs 2025 et avant).
Pour les CRs 2026+, voir notion-structure.md.

## Contexte à lire en premier
- reference/drive-structure.md → IDs des dossiers CA par année

## Étapes

### 1 — Identifier le dossier CA de l'année
Lire reference/drive-structure.md.
Prendre l'ID du dossier de l'année demandée
(ex. 2025 → 1BYIZfmSNz-FZwmdh2RfggnFOuf28tEpM).

### 2 — Lister les CR CA dans ce dossier
search_files({
  query: "parentId = 'ID_DU_DOSSIER'",
  orderBy: "modifiedTime desc",
  pageSize: 5
})
→ Identifier le document le plus récent avec "CA" dans le titre.

### 3 — Lire le contenu
read_file_content({ fileId: ID_DU_FICHIER_TROUVE })

### 4 — Extraire les décisions
Identifier dans le texte :
- Les décisions prises (votes, validations)
- Les actions assignées à des personnes
- Les points ouverts ou reportés au prochain CA

### 5 — Résumer
Format :
📋 CR CA du [date]

✅ Décisions :
- [liste]

👤 Actions assignées :
- [liste]

⏭️ Points reportés :
- [liste]

Ce skill illustre le pattern complet : lire la reference doc, chercher dans Drive avec un ID de dossier (pas un ID de fichier), lire le fichier trouvé, raisonner, résumer.

6. Combiner Drive et Notion dans un même skill

Le cas le plus courant n'est pas "Drive ou Notion" — c'est les deux. Un skill de brief complet pourrait :

  1. Lire les tâches Notion (notion-query-database-view)
  2. Lire le dernier CR CA Drive pour les décisions du CA
  3. Lire les créneaux bénévoles Drive pour le planning de la semaine
  4. Lire les messages Slack de la semaine (slack_read_channel)
  5. Composer un brief unifié avec toutes ces sources

C'est exactement ce que la leçon 5 a posé comme cible. Maintenant tu as la troisième jambe : Drive. Le skill brief-hebdo-actif peut être enrichi pour inclure cette lecture.

Ordre d'exécution conseillé dans un skill multi-sources
Fetch en parallèle quand possible (Notion + Slack + Drive). Raisonner ensuite sur l'ensemble. Agir (Slack, Notion) uniquement après avoir tout lu. Ne jamais agir avant de fetcher — c'est le principe de base depuis la leçon 1.

Ce qu'on a appris dans cette leçon

  • Drive vs Notion — les CRs CA récents sont dans Notion, les anciens dans Drive, le planning bénévoles dans Drive, les tâches dans Notion
  • Les trois outils Drive MCP — search_files, get_file_metadata, read_file_content
  • La syntaxe de requête — parentId =, title contains, mimeType =, combinables avec and
  • Le pattern "search before read" — chercher dans le dossier parent, puis lire le fichier trouvé
  • reference/drive-structure.md — la troisième reference doc, avec les IDs des dossiers CA et Gouvernance
  • Fetch multi-sources — Notion + Drive + Slack peuvent être lus dans un seul skill avant de raisonner
