Leçon 14 · Claude Code × MZD

Premier skill bénévole

Quand le destinataire n'est plus toi — concevoir un skill dont l'output est le produit.

Où on en est

Depuis la leçon 7, tu as appris à écrire des skills robustes pour toi, en tant qu'opérateur : fetch parallèle, gestion des erreurs, leading words, critères de complétion, failure modes, choix d'invocation.

La mission MZD n'est pas "automatiser la coordination de Gaspard". Elle est "rendre la MZD lisible et vivante pour ses bénévoles". Le premier skill vraiment aligné sur cette mission ne produit pas un brief pour toi — il produit un document pour quelqu'un qui découvre la MZD.

On va construire fiche-pole : un skill qui compose une fiche de présentation d'un pôle MZD, lisible par un bénévole qui ne connaît ni Notion, ni Slack, ni les acronymes internes.

1. Ce qui change quand le destinataire est un bénévole

Les patterns techniques restent les mêmes. Ce qui change, c'est le centre de gravité de la conception.

Skill opérateur (brief-hebdo-actif)Skill bénévole (fiche-pole)
Destinataire Toi — qui connais les outils et le contexte Un bénévole — qui découvre la MZD
Format de sortie Structuré pour toi (sections, 🔴/🟡/🟢, lien Notion) Lisible sans connaissance interne — aucun jargon, aucun ID
Ton Dense, technique, décisionnel Accessible, concret, orienté "comment rejoindre"
Ce qui est invisible Rien — les sources sont citées (section Sources) Tout — Notion/Slack ne sont jamais nommés comme "outils"

Dans un skill bénévole, l'output est le produit. La fiche que Claude compose n'est pas un artefact intermédiaire — c'est ce que le bénévole reçoit. Son format et son ton deviennent des contraintes de conception au même titre que les sources ou les barrières.

2. Le ton comme leading word

La leçon 10 a montré que les leading words recrutent des priors du préentraînement. Ça s'applique aussi au ton d'écriture.

Instruction de ton en prose
"Rédige de façon simple et compréhensible pour quelqu'un qui n'est pas familier avec les outils et le vocabulaire interne de l'association. Évite les termes techniques. Utilise des phrases courtes."

5 lignes. Claude va interpréter "simple" à sa manière.
Leading word de ton
"Format accessible — destinataire = bénévole qui découvre la MZD, aucune connaissance des outils internes."

"Accessible" recrute un prior précis : sans jargon, phrases courtes, orienté action. Claude sait ce que ça veut dire — il en a vu des milliers d'exemples.

La contrainte clé : aucun outil ne doit apparaître en tant que tel dans l'output. "Rejoint le canal Slack" devient "rejoint l'espace de discussion". "Page Notion" devient "la documentation". Le bénévole n'a pas besoin de savoir qu'on utilise Notion pour bénéficier de l'information.

3. Découverte dynamique vs ID hardcodé

Pour brief-hebdo-actif, tu fetches la base "Tâches MZD" avec un ID connu, stocké dans reference/notion-structure.md. Pour fiche-pole, tu n'as pas encore les IDs des pages de pôles dans ton reference file.

Il y a deux façons de gérer ça :

ApprocheQuand l'utiliserLimite
notion-fetch avec ID Page stable, souvent fetchée — ID dans reference/notion-structure.md Nécessite d'avoir enrichi le reference file à l'avance
notion-search avec le nom Page découverte dynamiquement — on ne connaît pas l'ID à l'avance Peut retourner plusieurs résultats — prendre le plus pertinent

Pour fiche-pole, on utilise notion-search avec le nom du pôle. C'est plus flexible — le skill fonctionne même si les IDs ne sont pas encore dans le reference file. Et ça révèle ce qu'il faudra éventuellement enrichir.

Règle pratique
Un skill qui fonctionne révèle ce qui manque dans l'infrastructure. fiche-pole montrera quelles pages Notion sont bien organisées et lesquelles sont introuvables. C'est de l'information — pas un bug.

4. fiche-pole — le SKILL.md

Voici le skill complet. Repère les patterns connus. 

---
name: fiche-pole
description: >
  Compose a readable pôle fact sheet for MZD volunteers — search the
  pôle's Notion page, fetch its Slack activity, format for non-technical
  audiences. Use when onboarding a volunteer or answering
  "what does pôle X do?"
---

## Contexte à lire avant de commencer

- `reference/slack-channels.md` — canaux Slack des pôles (IDs)

## Étape 0 — Identifier le pôle

Le pôle est nommé dans la demande.
Identifier son canal Slack dans reference/slack-channels.md.
Si aucun canal correspondant → noter "Slack : aucun canal trouvé", continuer.

## Étape 1 — Fetch (en parallèle)

a) `notion-search` avec le nom du pôle
   → Prendre la page dont le titre correspond le mieux
   Si aucun résultat → noter "Notion : page pôle introuvable", continuer

b) `slack_read_channel` sur le canal du pôle
   → ID dans reference/slack-channels.md
   Pour les 14 derniers jours
   Si silencieux → noter "canal silencieux depuis 14 jours", continuer
   Si erreur → noter "Slack : indisponible", continuer

Continue quand toutes les tentatives sont terminées,
qu'elles aient réussi ou non.

## Étape 2 — Composer la fiche

Format accessible — destinataire = bénévole qui découvre la MZD,
aucune connaissance des outils internes.

**[Nom du pôle]**
_[Mission en une phrase — ce que fait le pôle concrètement]_

**Ce qu'on fait**
[2-3 activités concrètes — sans jargon, sans noms d'outils]

**Comment s'investir**
[Action concrète : "rejoins l'espace de discussion #..." ou "contacte [prénom]"]

**Ce qui se passe en ce moment**
[Résumé de l'activité des 14 derniers jours — sans IDs ni références techniques
 Si silencieux → "Le pôle est peu actif en ce moment."]

**Pour aller plus loin**
[Où trouver les infos — canal + documentation — présentés sans nommer les outils]

Règle : aucun acronyme non défini, aucune mention de Notion/Slack comme
"outils", aucun ID visible. Notion = "la documentation", Slack = "l'espace
de discussion".

Continue quand la fiche est complète et lisible sans connaissance interne.

5. Les patterns que tu reconnais

Rien de nouveau dans ce skill — seulement des patterns connus, réassemblés pour un nouveau but.

  • Étape 0 — reference file comme source de vérité. Même pattern que brief-hebdo-actif.
  • Fetch parallèle + repli par source — leçons 8 et 9. Chaque source a son instruction de repli. Le skill avance même si Notion est introuvable.
  • Barrière robuste — "tentatives terminées, qu'elles aient réussi ou non" — leçon 11.
  • Leading words — "accessible" pour le ton, "compose" dans la description — leçon 10.
  • User-invoked — pas de side effects ici (lecture + composition), mais usage délibéré — leçon 13. Tu l'appelles quand un bénévole te demande "c'est quoi le pôle X ?".

6. Ce que le skill va révéler en pratique

La première fois que tu utilises fiche-pole sur le pôle boutique, tu vas probablement découvrir l'une de ces situations :

  • Notion : page introuvable — le pôle n'a pas de page structurée dans Notion. La fiche sera basée seulement sur Slack. C'est de l'information : cette page est à créer.
  • Slack : canal silencieux — le canal du pôle n'a pas d'activité récente. La fiche le mentionne honnêtement. C'est aussi de l'information : le pôle est peut-être inactif.
  • Les deux disponibles — la fiche est riche. Tu sais que l'infrastructure de ce pôle est en ordre.

Un skill bénévole est aussi un outil de diagnostic de l'organisation : il révèle quels pôles sont documentés, lesquels sont actifs, et lesquels ont disparu de la surface visible de la MZD.

Ce qu'on a appris dans cette leçon

  • L'output est le produit — dans un skill bénévole, le format et le ton de la sortie sont des contraintes de conception au même titre que les sources
  • Le ton comme leading word — "accessible" et "destinataire = bénévole qui découvre la MZD" recrutent le bon registre d'écriture sans prose explicative
  • notion-search pour la découverte dynamique — quand les IDs ne sont pas encore dans le reference file, notion-search permet de démarrer sans blocage
  • Un skill révèle l'état de l'organisation — les pages introuvables et les canaux silencieux sont de l'information, pas des erreurs
  • Tous les patterns techniques restent — fetch parallèle, repli par source, barrière robuste, leading words — rien de nouveau, juste un nouveau destinataire
