Leçon 6 · Claude Code × MZD

Autonomie planifiée

Les crons — programmer les skills pour qu'ils se déclenchent seuls, et savoir exactement où garder la main humaine.

Où on en est

Tu as un système complet : des skills qui fetchent, raisonnent, agissent, et des reference docs qui centralisent la structure de la MZD. Le seul point d'entrée qui reste humain, c'est le déclencheur.

Chaque vendredi, quelqu'un doit se souvenir de lancer /cloture-semaine vers 17h. Si personne ne le fait — le brief n'est pas produit, le message ne part pas.

La solution s'appelle un cron.

1. Les crons Claude Code

Un cron est une tâche planifiée : à une heure précise, Claude reçoit un prompt et l'exécute, comme si tu l'avais tapé toi-même. Il utilise les mêmes outils, lit les mêmes fichiers, appelle les mêmes MCP.

Tu le crées avec CronCreate, qui prend deux paramètres clés :

  • cron — l'heure d'exécution en syntaxe cron standard (5 champs : minute heure jour mois jour-semaine)
  • prompt — le texte à envoyer à Claude à cette heure-là 
CronCreate({
  cron: "3 17 * * 5",      // chaque vendredi à 17h03
  prompt: "Exécute le skill cloture-semaine. \
Lis d'abord reference/slack-channels.md et \
reference/notion-structure.md.",
  durable: true            // persiste entre les sessions
})

0 17 * * 5 signifie : minute 0, heure 17, n'importe quel jour du mois, n'importe quel mois, jour 5 de la semaine (vendredi). Par convention, on décale légèrement (ici 3) pour éviter les pics de charge quand tout le monde programme la même heure ronde.

Les deux modes

ParamètreValeurComportement
durable: false défaut En mémoire — disparaît quand la session Claude se ferme
durable: true persistant Écrit dans .claude/scheduled_tasks.json — survit aux redémarrages
recurring: true défaut Se répète à chaque correspondance. Auto-expire après 7 jours.
recurring: false unique Se déclenche une seule fois puis se supprime
Limite importante
Les crons ne se déclenchent que si Claude Code est en cours d'exécution et la session est active. Si tu fermes l'application, les crons non-durables disparaissent ; les durables reprendront au prochain démarrage. Et les récurrents expirent automatiquement après 7 jours — il faut les recréer.

2. Écrire un bon prompt de cron

Un prompt de cron est différent d'une invocation manuelle. Quand tu tapes /cloture-semaine, tu as du contexte : tu sais que c'est vendredi, tu sais que l'équipe attend le message. Claude n'a rien de tout ça — il part d'une page blanche.

Prompt fragile
"Lance la clôture de semaine."

Claude ne sait pas quel skill. Il ne sait pas quels fichiers lire. Il risque d'inventer ou d'échouer silencieusement.
Prompt robuste
"C'est vendredi vers 17h. Exécute le skill cloture-semaine défini dans .agents/skills/cloture-semaine/SKILL.md. Lis d'abord reference/slack-channels.md et reference/notion-structure.md."

Un bon prompt de cron contient trois choses :

  1. Le contexte temporel — quelle heure, quel jour, pourquoi maintenant
  2. Le skill à exécuter — chemin exact du SKILL.md
  3. Les fichiers de contexte à lire — reference docs indispensables

3. Où garder la main humaine

Automatiser un skill, c'est retirer l'humain du déclencheur. Mais ce n'est pas la même chose que de retirer l'humain des décisions.

Voici comment classer les actions par niveau de risque :

Type d'actionExemplesAutomatisable ?
Lecture seule Fetch Notion, lire Slack, analyser ✅ Oui, sans hésiter
Brouillon Composer un message, préparer un ticket ✅ Oui — c'est juste une préparation
Envoi réversible Message Slack (peut être supprimé) ⚠️ Oui si tu fais confiance au skill
Écriture Notion Créer un ticket, modifier une page ⚠️ Avec idempotence, oui
Action irréversible Archiver, supprimer, notifier toute l'équipe ❌ Garder la confirmation humaine

Le pattern "brouillon d'abord"

Pour /cloture-semaine, le skill actuel dit déjà "montre avant de poster". Dans un cron sans humain qui regarde, cette confirmation ne peut pas fonctionner comme prévu.

La solution : utiliser slack_send_message_draft au lieu de slack_send_message. L'outil existe dans Slack MCP et crée un brouillon que tu vois dans Slack avant d'envoyer. 

## Étape 4 — Envoyer sur Slack (version cron)

Si exécuté par un cron (pas d'humain qui regarde) :
  Appelle slack_send_message_draft sur #échanges-ca-équipe.
  → Le brouillon apparaît dans Slack pour révision humaine.

Si exécuté manuellement :
  Montre le message. Attends confirmation. Puis slack_send_message.
Règle à retenir
Automatise les opérations (fetch, analyse, composition). Garde la main humaine sur les décisions (envoyer, créer, modifier). Un cron qui produit un brouillon + une validation rapide dans Slack est souvent meilleur qu'un cron entièrement autonome.

4. Syntaxe cron — les cas MZD

┌─────── minute (0–59)
│ ┌───── heure (0–23)
│ │ ┌─── jour du mois (1–31)
│ │ │ ┌─ mois (1–12)
│ │ │ │ └ jour de la semaine (0=dim, 1=lun … 5=ven, 6=sam)
│ │ │ │ │
3 17 * * 5    → chaque vendredi à 17h03  (/cloture-semaine)
7  9 * * 1    → chaque lundi à 9h07      (/brief-hebdo)
0  8 * * 1-5  → chaque jour de semaine à 8h  (rappel)
0 10 1 * *    → le 1er de chaque mois à 10h

5. Créer le cron /cloture-semaine

Voici l'appel complet pour programmer la clôture de semaine automatique : 

CronCreate({
  cron: "3 17 * * 5",
  prompt: `C'est vendredi vers 17h — heure de clôture de semaine MZD.

Exécute le skill défini dans :
  .agents/skills/cloture-semaine/SKILL.md

Avant de commencer, lis :
  reference/slack-channels.md
  reference/notion-structure.md

À l'étape 4 (envoi Slack), envoie un brouillon avec
slack_send_message_draft plutôt que slack_send_message,
pour permettre une révision humaine avant publication.`,
  durable: true,
  recurring: true
})

Ce cron crée un fichier .claude/scheduled_tasks.json dans ton repo — il survit aux redémarrages de Claude Code. Il se répète chaque vendredi et s'auto-expire après 7 jours (à recréer ensuite).

Consulter et supprimer les crons

CronList()           // voir tous les crons actifs
CronDelete("job-id") // supprimer un cron par son ID

Ce qu'on a appris dans cette leçon

  • CronCreate — planifier un prompt récurrent avec syntaxe cron standard (minute heure jour mois dow)
  • Prompts de cron robustes — contexte temporel + chemin du skill + reference docs à lire
  • Le pattern brouillon — automatiser la production, garder la main humaine sur l'envoi via slack_send_message_draft
  • durable: true — pour que le cron survive aux redémarrages (écrit dans .claude/scheduled_tasks.json)
  • 7-day expiry — les crons récurrents s'auto-expirent, il faut les recréer chaque semaine (ou les déclencher manuellement)
