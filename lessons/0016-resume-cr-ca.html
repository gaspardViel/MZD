Résumés de CR CA

Transformer un compte-rendu administratif en contenu lisible par les bénévoles — sans perdre la fidélité à la source.

Où on en est

En leçon 15, tu as construit gouvernance-partager — un skill qui fetche la structure MZD depuis Notion et produit un document de référence durable pour les bénévoles. Tu connais la différence entre données stables et dynamiques, et entre les patterns triage et orient.

Il reste un bloc de la mission : les résumés de CR CA. Les comptes-rendus du Conseil d'Administration existent dans Notion ou Drive. Mais ils sont écrits en langage administratif, pour un usage juridique et interne. Les bénévoles n'y ont pas facilement accès — et même quand ils y ont accès, ils ne savent pas quoi en retenir.

1. Le problème du document formel

Un CR de CA est un document légal. Il consigne les décisions, les votes, les présences, les points à l'ordre du jour. Il est écrit pour être juste — pas pour être lisible.

Ce qui intéresse un bénévole qui lit un CR CA :

  • Qu'est-ce qui a été décidé qui me concerne ?
  • Pourquoi cette décision a été prise ?
  • Est-ce que quelque chose va changer dans mon pôle ou ma façon de travailler ?

Ce que le CR CA dit à la place :

  • "Le CA a voté à l'unanimité la résolution n°3 relative à l'affectation budgétaire du pôle X."
  • "Point n°7 — informations diverses : [néant]."
  • Listes de présence, signatures, quorum.

L'écart entre les deux n'est pas un défaut du document — c'est sa fonction. Le skill resume-cr-ca comble cet écart.

2. La transformation documentaire

C'est un troisième pattern, distinct du triage et de l'orient :

Pattern Source Output Skill
Triage Données en cours (tâches, messages Slack) Rapport opérationnel éphémère brief-hebdo-actif
Orient Structure organisationnelle dans Notion Document de référence durable gouvernance-partager
Transformation Document formel (CR CA, PV) Résumé accessible pour un public différent resume-cr-ca

Ce qui distingue la transformation des deux autres patterns : la source a un public cible différent de l'output. Le CR CA est écrit pour le CA. Le résumé est écrit pour les bénévoles. Ce n'est pas une reformulation — c'est un changement de public.

3. Les deux tensions

Ce changement de public crée deux tensions à gérer explicitement.

Trop fidèle
Le résumé reprend le langage administratif du CR :

"Le CA a délibéré sur la résolution relative à l'affectation du budget pôle communication et a voté favorablement à l'unanimité."

Résultat : illisible pour un bénévole. Aucune valeur ajoutée.
Bien transformé
Le résumé traduit en langage bénévole :

"Le CA a validé le budget communication pour le trimestre. Le pôle peut lancer les actions prévues sans attendre une nouvelle validation."

Résultat : actionnable, lisible, utile.
Trop interprété
Le skill infère des intentions qui ne sont pas dans le CR :

"Le CA semble vouloir réorienter la stratégie communication vers les réseaux sociaux."

Résultat : hallucination éditoriale. Ce n'est pas dans le CR. Les bénévoles lisent quelque chose de faux.
Bien borné
Le skill ne dit que ce qui est dans le CR :

"Budget communication validé. Aucune mention de priorités spécifiques dans le CR — pour les détails, contacter le référent pôle communication."

Résultat : fidèle à la source, honnête sur ses limites.

La règle pour naviguer entre les deux :

Traduis le langage, préserve les faits

Tu peux simplifier le vocabulaire, raccourcir les formulations, changer la structure. Tu ne peux pas inférer des intentions, ajouter du contexte absent du CR, ni omettre une décision parce qu'elle semble peu importante. Tout ce qui est dans le CR peut aller dans le résumé. Rien de ce qui n'y est pas ne peut y entrer.

4. Le filtre de pertinence

Un CR CA contient des points qui n'intéressent pas les bénévoles : approbation des comptes, quorum, signatures, points de forme juridique. Le skill doit filtrer — mais ce filtre doit être explicite, pas silencieux.

Filtre silencieux = danger

Si le skill omet un point sans le signaler, le bénévole croit avoir lu tout le CR. Il manque peut-être une décision qui le concerne. Le filtre doit être déclaré dans le résumé : "Points administratifs non repris ici (quorum, signatures) — CR complet disponible [lien]."

Structure de résumé en trois blocs :

  1. Décisions prises — liste des décisions avec leur impact concret pour les bénévoles. Une phrase par décision.
  2. Contexte clé — éléments de discussion retenus qui aident à comprendre pourquoi la décision a été prise. Seulement ce qui est explicitement dans le CR.
  3. Pied de page — date du CA, lien vers le CR complet, mention des points administratifs non repris.

5. Conception du skill

---
name: resume-cr-ca
description: >
  Summarize a CA meeting (compte-rendu) into a volunteer-facing digest —
  fetch the CR from Notion or Drive, extract decisions and key context,
  translate administrative language into plain French, and post to Slack
  or Notion.
  Run after each CA meeting once the CR is published.
argument-hint: "[lien ou ID du CR à résumer]"
---

## Objectif

Produire un résumé en trois blocs du dernier CR de CA — lisible par
un bénévole qui n'était pas présent, fidèle à la source, sans inférence.

## Étape 0 — Charger les références

Lire `reference/notion-structure.html` et/ou `reference/drive-structure.html`
pour localiser les CRs de CA (dossier Drive ou base Notion dédiée).
Si un ID ou lien est fourni en argument, utiliser cet ID directement.

## Étape 1 — Fetcher le CR

Récupérer le contenu complet du CR depuis Notion ou Drive.
Si le document est absent ou inaccessible, arrêter :
"CR introuvable à cet emplacement. Vérifier l'ID fourni ou
mettre à jour reference/notion-structure.html."

## Étape 2 — Extraire et filtrer

Identifier dans le CR :
- Les décisions votées (résolutions, validations, refus)
- Les éléments de contexte explicitement mentionnés
- Les points administratifs à mentionner mais non développer
  (quorum, signatures, approbation PV précédent)

Ne pas inférer. Si un point est ambigu, le retranscrire tel quel
entre guillemets plutôt que de l'interpréter.

## Étape 3 — Rédiger le résumé

**Bloc 1 — Décisions**
Pour chaque décision : une phrase, langage courant, impact concret.
Exemple : "Budget événement estival validé (3 000 €)."

**Bloc 2 — Contexte clé**
2-4 phrases maximum. Seulement ce qui est dans le CR.
Si aucun contexte utile n'est mentionné : omettre ce bloc.

**Bloc 3 — Pied de page**
"CA du [date]. Points administratifs non repris (quorum, signatures).
CR complet : [lien]. Pour questions : [contact référent CA si disponible]."

## Étape 4 — Publier

Si canal Slack fourni : poster le résumé.
Si page Notion fournie : créer une sous-page dans la base CRs.
Sinon : afficher et demander confirmation avant publication.

## Critère de complétion

✓ Toutes les décisions du CR sont reprises dans le Bloc 1
✓ Aucune information absente du CR n'est présente dans le résumé
✓ Le pied de page mentionne les points administratifs omis et le lien source
✓ Un bénévole peut comprendre l'impact sans lire le CR complet

Quiz

Le CR mentionne une discussion sur la stratégie de communication sans conclure par une décision formelle. Que fait le skill ?

Le CR contient 12 points dont 4 décisions et 8 points administratifs (approbation du PV, quorum, signatures). Que fait le résumé des 8 points ?

Quelle est la différence principale entre le pattern "orient" (gouvernance-partager) et le pattern "transformation" (resume-cr-ca) ?

Ce qu'on a appris dans cette leçon

  • Pattern transformation — convertir un document formel pour un public différent de son public source. Distinct du triage (données en cours) et de l'orient (structure organisationnelle).
  • Traduis le langage, préserve les faits — simplification du vocabulaire autorisée, inférence interdite. Tout ce qui n'est pas dans le CR ne peut pas entrer dans le résumé.
  • Filtre déclaré, pas silencieux — les points administratifs omis doivent être mentionnés dans le pied de page avec un lien vers la source complète.
  • Structure en trois blocs — décisions, contexte clé, pied de page. Le bloc contexte est optionnel si rien d'utile n'est explicitement mentionné dans le CR.

Source recommandée

La prochaine étape concrète : localiser un CR de CA réel dans Notion ou Drive et exécuter resume-cr-ca sur ce document. C'est là que la règle "traduis le langage, préserve les faits" devient tangible — la tension entre fidélité et accessibilité n'est visible qu'avec un vrai document source.

Une question ? Demande à ton agent. "Comment trouver les CRs de CA dans notre Notion ?" ou "Comment gérer un CR qui ne liste pas clairement les décisions ?" sont de bonnes questions de départ.
